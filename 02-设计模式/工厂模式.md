# 工厂模式

日期：2026-07-07

## 1. 工厂模式为什么出现
1. 如果没有工厂，业务代码会怎么创建对象？
在各自的业务方法创建对象
2. 到处 new 对象、写 if else 会带来什么问题？
当遇到需要新增或修改时，就要到处改
3. 工厂模式解决了什么问题？
统一创建对象，避免在业务代码创建对象

## 2. 简单工厂模式是什么
1. 简单工厂由哪些角色组成？
接口、具体实现类、工厂类和调用方
2. 调用方如何通过工厂获取对象？
调用工厂的创建方法，根据参数判断应该创建什么对象
3. 简单工厂是否属于 23 种经典设计模式？
不属于，简单工厂不符合开闭原则，即对扩展开放，对修改关闭。

## 3. 简单工厂模式的优点
1. 对象创建逻辑集中管理。
2. 减少业务代码中的重复 if else。
3. 调用方不需要关心具体实现类。
4. 业务代码更清晰。

## 4.简单工厂模式的缺点
1. 新增产品类型时，需要修改工厂类。
2. 产品类型越来越多时，工厂类会变得臃肿。
3. 不完全符合开闭原则。

## 5.适合使用的场景
1. 对象类型不多。
2. 创建逻辑比较集中。
3. 调用方不想关心具体实现类。
4. 新增类型频率不高。

## 6.面试总结
简单工厂模式是为了解决对象创建逻辑散落在业务代码中的问题。它通过一个工厂类，根据传入的类型创建不同的具体对象，调用方只依赖统一接口，不需要关心具体实现类。
它的优点是创建逻辑集中、业务代码更清晰；缺点是新增类型时需要修改工厂类，因此不完全符合开闭原则。适合对象类型不多、创建逻辑相对稳定的场景。
