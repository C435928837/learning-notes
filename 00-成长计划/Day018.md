## 2. CompletableFuture 是什么

我的理解：

>CompletableFuture可以执行任务，可以返回执行任务后的结果，同时还为多个任务提供编排异步流程功能

与之前线程相关类型的关系：

| 类型 | 本质                    | 是否有返回值 | 作用             |
| --- |-----------------------|--------|----------------|
| `Runnable` | 执行任务                  | 否      | 只执行任务          |
| `Future` | 获取任务结果                | 有      | 获取结果           |
| `FutureTask` | 执行任务并返回结果             | 有      | 执行任务和获取结果      |
| `CompletableFuture` | 执行任务并返回结果，还可以对异常情况做处理 | 有      | 执行任务、获取结果、处理异常 |

核心结论：

>`CompletableFuture` 不只是获取异步结果，更重要的是可以对异步结果或异常情况做进一步的处理。`

## 3. 创建异步任务

### 3.1 runAsync()

```java
CompletableFuture<Void> future =
        CompletableFuture.runAsync(() -> {
            // 无返回值任务
        }, executor);
```

适用场景：

>无需等待结果

返回类型：

>CompletableFuture<Void>

### 3.2 supplyAsync()

```java
CompletableFuture<String> future =
        CompletableFuture.supplyAsync(
                () -> queryProduct(productId),
                executor
        );
```

适用场景：

>需要返回结果

返回类型：

>CompletableFuture<T>

为什么业务代码应传入自定义线程池：

>completable使用的默认线程池是JVM的公共线程池，不符合业务的性能场景，多线程的话会带来阻塞

未传入线程池时，通常使用：

>ForkJoinPool.commonPool

## 4. 异步编排

### 4.1 thenApply()

```java
future.thenApply(result -> {
    return transform(result);
});
```

作用：

>为返回结果做下一步处理

适用场景：

>需要对返回结果做下一步处理的场景

### 4.2 thenCompose()

```java
future.thenCompose(result ->
        CompletableFuture.supplyAsync(
                () -> nextAsyncQuery(result),
                executor
        )
);
```

作用：

>对返回结果做下一步任务调用

与 `thenApply()` 的区别：

>thenApply()只做处理，不调用任务

### 4.3 thenCombine()

```java
productFuture.thenCombine(
        stockFuture,
        (product, stock) -> buildDetail(product, stock)
);
```

作用：

>把两个任务的返回结果做合并

适用场景：

>需要合并两个任务的结果的场景

### 4.4 allOf()

```java
CompletableFuture.allOf(
        productFuture,
        stockFuture,
        promotionFuture
);
```

作用：

>等待所有任务完成

`allOf()` 的返回类型：

>全部完成后应通过各 Future 的 join() 获取结果，或在 thenApply() 中组合结果

## 5. 异常处理与等待

### 5.1 exceptionally()

作用：

>对任务异常情况做兜底处理

适用场景：

>降级

### 5.2 handle()

作用：

>无论任务正不正常结束都做处理

与 `exceptionally()` 的区别：

>exceptionally()仅异常时执行，handle()成不成功都执行

### 5.3 whenComplete()

作用：

>观察结果或记录日志，不负责恢复

适用场景：

>日志记录、埋点

### 5.4 get() 与 join()

| 对比项 | `get()` | `join()` |
| --- |---------|----------|
| 是否等待任务完成 | 是       | 是        |
| 异常类型 |    InterruptedException、ExecutionException     |    CompletionException，运行时异常      |
| 是否需要强制捕获异常 | 是       | 否        |
| 是否支持超时参数 | 支持      | 不支持      |

我的选型结论：

>需要明确处理超时、受检异常的场景选择get()，在 CompletableFuture 的链路末端快速取结果选择join()

## 7. 高频面试题

### 7.1 CompletableFuture 解决了什么问题？

>异步任务的串行并行处理逻辑

### 7.2 thenApply、thenCompose、thenCombine 的区别？

>thenApply对返回结果做处理，thenCompose把返回结果作为参数调用下一个任务，thenCombine合并两个任务的结果

### 7.3 allOf 的作用是什么？如何获取各任务结果？

>全部完成后应通过各 Future 的 join() 获取结果，或在 thenApply() 中组合结果

### 7.4 为什么异步业务要使用自定义线程池？

>自定义线程池能为各种业务场景提供更有效的线程安排和线程排查，如果用默认线程池，所有业务的线程都会在一个线程池，容易发生阻塞。

### 7.5 CompletableFuture 的异常如何处理？

>可以用exceptionally做降级处理，也可以用handle做统一兜底处理
