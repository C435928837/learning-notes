## 2. 线程池是什么

线程池的本质：

预先创建并管理一批可复用线程，用任务队列接收任务，由空闲线程持续执行任务

线程池解决的问题：

- 频繁创建线程
- 任务瞬间过多
- 缺少统一关闭

为什么不建议频繁 `new Thread()`：

线程的创建和销毁都需要系统资源的开销，大量创建线程会占用系统资源，无法统一线程的状态、异常情况和拒绝策略。

CloudMall 场景：

用户打开商品详情页时，商品信息、库存、优惠信息可以并行查询；批量更新商品状态、异步发送通知。

## 3. ThreadPoolExecutor 核心参数

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
        4,
        8,
        60,
        TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(200),
        threadFactory,
        new ThreadPoolExecutor.CallerRunsPolicy()
);
```

| 参数 | 作用           | CloudMall 场景 |
| --- |--------------| --- |
| `corePoolSize` | 核心线程数        | 商品异步任务平稳到来时，通常由 4 个线程处理，避免频繁创建线程 |
| `maximumPoolSize` | 最大线程数        | 商品批量更新或活动开始时任务突增，队列满后最多扩展到 8 个线程临时处理高峰 |
| `keepAliveTime` | 非核心线程空闲多久后回收 | 秒杀或促销高峰结束后，临时扩容出的线程空闲 60 秒会被回收，避免长期占用资源 |
| `TimeUnit` | 时间单位         | 表示上述 60 的单位是秒，实际可按任务特点选择秒或分钟 |
| `workQueue` | 任务队列         | 最多允许 200 个商品异步任务排队，防止任务无限堆积导致内存压力 |
| `ThreadFactory` | 线程工厂         | 线程名为 cloudmall-product-async-*，出现阻塞、异常时可通过日志或 jstack 快速定位商品线程池 |
| `RejectedExecutionHandler` | 拒绝策略         | 系统过载时让提交方变慢，形成反压，避免直接丢失商品变更任务。若是 HTTP 主线程场景，要谨慎使用，可能拖慢接口响应 |

我的参数选型结论：

根据 CPU 核数、任务是 CPU 密集还是 I/O 密集、接口流量、任务耗时和压测结果调整

## 4. 线程池执行流程

```text
当前线程数 < corePoolSize 创建核心线程；核心线程满后先入队；队列满且当前线程数 < maximumPoolSize 才创建非核心线程；队列满且线程数 >= 最大值时触发拒绝策略。
```

核心结论：

> 最大线程数不是核心线程满后立刻创建，而是队列满后才会尝试创建非核心线程。

## 5. 阻塞队列对比

| 队列 | 底层结构       | 容量特点       | 线程池中的行为               | 使用建议          |
| --- |------------|------------|-----------------------|---------------|
| `ArrayBlockingQueue` | 数组         | 必须制定长度     | 队列未满时优先排队             |    需要明确限流、控制内存           |
| `LinkedBlockingQueue` | 链表         | 可指定，默认近似无限 | 容易持续排队，最大线程数可能难以生效    |  生产消费任务较多，但必须设置容量             |
| `SynchronousQueue` | 不存储元素，直接交接 | 0          | 无法排队，优先快速创建线程，直到最大线程数 | 短任务、直接交接、快速扩容 |

## 6. 拒绝策略

| 策略 | 行为        | 适用场景          |
| --- |-----------|---------------|
| `AbortPolicy` | 抛出异常      |   适合重要任务。调用方捕获异常后记录日志、重试、降级或返回“系统繁忙”。            |
| `CallerRunsPolicy` | 由当前线程执行   |   形成反压，避免直接丢任务。若提交方是 HTTP 请求线程，可能拖慢接口响应。            |
| `DiscardPolicy` | 直接放弃当前任务  | 仅适合允许丢失的非核心任务 |
| `DiscardOldestPolicy` | 放弃最早排队的任务 | 仅关心最新状态的非核心任务 |

订单、库存等核心任务为什么不适合静默丢弃：

静默丢弃会造成订单、库存的信息丢失，是非常严重的问题

## 7. Executors 与 ThreadPoolExecutor

`ExecutorService`、`ThreadPoolExecutor`、`Executors` 的关系：

| 名称 | 本质  | 作用                       |
| --- |-----|--------------------------|
| `ExecutorService` | 接口  | 定义提交任务、关闭线程池、获取Future等方法 |
| `ThreadPoolExecutor` | 实现类 | 真正实现核心线程、最大线程、队列、拒绝策略等逻辑 |
| `Executors` | 工具类 | 提供快速创建线程池的方法             |

为什么不建议直接使用 `Executors`：

创建的固定线程池和单线程池默认使用无界队列，可能导致任务堆积和 OOM；缓存线程池最大线程数接近无限，高并发下可能创建大量线程并耗尽资源

## 10. 面试题整理

### 10.1 为什么不建议频繁 new Thread()？

答：线程的创建与销毁都需要系统资源的开销，大量创建线程会占用系统大量空间，无法统一控制线程的数量、状态和异常情况。

### 10.2 ThreadPoolExecutor 的执行流程是什么？

答：当前线程数 < corePoolSize 创建核心线程；核心线程满后先入队；队列满且当前线程数 < maximumPoolSize 才创建非核心线程；队列满且线程数 >= 最大值时触发拒绝策略。

### 10.3 核心线程、最大线程数和队列如何配合？

答：核心线程未满时先创建核心线程，核心线程满了任务进入队列，队列满了创建非核心线程，最大线程数满了则出发拒绝策略

### 10.4 常见拒绝策略有哪些？如何选择？

答：AbortPolicy：抛出异常；CallerRunsPolicy：由当前线程执行；DiscardPolicy：丢弃当前任务；DiscardOldestPolicy：丢弃排队最早的任务

### 10.5 为什么不建议直接使用 Executors？

答：Executors的部分工厂方法创建线程方法有风险，newFixedThreadPool 和 newSingleThreadExecutor 使用近似无界队列；newCachedThreadPool 的风险是最大线程数接近无限
