## 2. 为什么需要工厂方法模式

简单工厂的问题是：创建逻辑都集中在一个工厂累

当新增一种商品类型时，需要修改：工厂类

这会带来的问题是：违反开闭原则，当需要新增商品时就要改工厂类的代码

工厂方法模式的思路是：不同的商品由不同的工厂类创建，工厂类只需要实现工厂接口。

## 3. 核心概念

### 3.1 产品接口 Product

```java
public interface Product {
    String getName();
}
```

产品接口的作用：提供产品的业务方法。

### 3.2 具体产品

```java
public class NormalProduct implements Product {
    @Override
    public String getName() {
        return "普通商品";
    }
}
```

```java
public class SeckillProduct implements Product {
    @Override
    public String getName() {
        return "秒杀商品";
    }
}
```

具体产品负责：实现具体产品的业务方法。

### 3.3 工厂接口 ProductFactory

```java
public interface ProductFactory {
    Product createProduct();
}
```

工厂接口的作用：提供创建产品的接口。

### 3.4 具体工厂

```java
public class NormalProductFactory implements ProductFactory {
    @Override
    public Product createProduct() {
        return new NormalProduct();
    }
}
```

```java
public class SeckillProductFactory implements ProductFactory {
    @Override
    public Product createProduct() {
        return new SeckillProduct();
    }
}
```

具体工厂负责：创建具体的产品。

### 3.5 调用示例

```java
public class FactoryMethodDemo {
    public static void main(String[] args) {
        ProductFactory factory = new SeckillProductFactory();
        Product product = factory.createProduct();

        System.out.println(product.getName());
    }
}
```

运行结果：

```text
秒杀商品
```

这里 `Product` 是接口类型引用，实际指向的是：new SeckillProduct()

这体现了 Java 的：多态。

## 4. 简单工厂与工厂方法对比

| 对比项 | 简单工厂 | 工厂方法 |
| --- | --- | --- |
| 创建方式 | 一个工厂创建多种产品 | 每种产品对应一个具体工厂 |
| 新增产品 | 修改原工厂 | 新增产品类和工厂类 |
| 开闭原则 | 不完全符合 | 更符合 |
| 类数量 | 较少 | 较多 |
| 适用场景 | 产品类型较少、变化不大 | 产品扩展频繁 |

我的理解：简单工厂的创建逻辑都集中一个工厂类，当需要扩展的时候需要修改工厂类。工厂模式的创建逻辑在各自的工厂实现类，当需要拓展时只需要新增产品类和产品工厂类就行。

## 6. 算法题：只出现一次的数字

题目：给定一个非空整数数组，除了某个元素只出现一次以外，其余每个元素均出现两次，找出那个只出现一次的元素。

### 6.1 HashSet 解法

思路：判断元素是否在set集合中，存在就去除，不存在则新增

时间复杂度：O(n)  
空间复杂度：O(n)

### 6.2 异或优化

```java
class Solution {
    public int singleNumber(int[] nums) {
        int result = 0;

        for (int num : nums) {
            result ^= num;
        }

        return result;
    }
}
```

异或规律：

```text
a ^ a = 0
a ^ 0 = a
```

示例推导：

```text
[4, 1, 2, 1, 2]
= 0^4^1^2^1^2
= 4^(1^1)^(2^2)
= 4^0^0
```

时间复杂度：O(n)  
空间复杂度：O(1)

适用条件：需要判断是否存在重复元素

## 7. 面试题整理

### 7.1 什么是工厂方法模式？

答：是创建型设计模式，把具体对象的创建延迟到产品工厂类。产品工厂接口只负责提供创建产品的接口，具体的产品由对应的产品工厂实现类创建。

### 7.2 工厂方法模式和简单工厂有什么区别？

答：简单工厂的创建逻辑集中在一个工厂类，工厂模式的创建逻辑由不同的工厂实现类实现。

### 7.3 工厂方法模式的优缺点是什么？

答：优点：符合开闭原则，当需要新增产品时只需要新增产品类和产品工厂类就行。缺点：需要创建很多类。

### 7.4 为什么 `Product` 接口类型可以接收具体商品对象？

答：接口不能直接实例化，但接口类型的引用可以指向其实现类对象。例如 Product product = new SeckillProduct()。这样调用方依赖抽象，运行时可替换不同实现，这体现了多态。

### 7.5 “只出现一次的数字”为什么可以用异或解决？

答：相同数字异或结果为 0，0 与任何数字异或仍为该数字；异或满足交换律和结合律，因此成对元素会抵消，最后剩下只出现一次的元素。
