## 3. lock()、unlock() 与 finally

基础写法：

```java
private final ReentrantLock lock = new ReentrantLock();

public boolean deductStock() {
    lock.lock();
    try {
        // 判断库存并扣减
        return true;
    } finally {
        lock.unlock();
    }
}
```

`lock()` 的作用：

>获得锁

`unlock()` 的作用：

>释放锁

为什么必须在 `finally` 中释放锁：

>避免执行代码过程中抛出异常，没有释放锁

提前 `return` 时，`finally` 是否执行：

>会执行

未持有锁时调用 `unlock()` 会：

>抛出异常

## 4. 可重入性

示例：

```java
public void methodA() {
    lock.lock();
    try {
        methodB();
    } finally {
        lock.unlock();
    }
}

public void methodB() {
    lock.lock();
    try {
        // 执行业务
    } finally {
        lock.unlock();
    }
}
```

执行过程：

```text
methodA 第一次获得锁，持有次数：
methodB 再次获得同一把锁，持有次数：
methodB 释放锁后，持有次数：
methodA 释放锁后，持有次数：
```

可重入性的含义：

>已经获得锁的线程在调用方法时无需再竞争锁，获得多少次锁就要释放多少次锁

## 5. 尝试获取锁与可中断锁

### 5.1 tryLock()

```java
if (lock.tryLock()) {
    try {
        // 获取锁成功
    } finally {
        lock.unlock();
    }
} else {
    // 获取锁失败
}
```

特点：

>只尝试一次，获取失败不再尝试

适用场景：

>秒杀商品

### 5.2 超时获取锁

```java
boolean locked = false;

try {
    locked = lock.tryLock(2, TimeUnit.SECONDS);

    if (!locked) {
        return;
    }

    // 获取锁成功后的业务代码
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
} finally {
    if (locked) {
        lock.unlock();
    }
}
```

特点：

>会在指定时间内尝试获得锁，超时还未获得锁就不再尝试

### 5.3 lockInterruptibly()

```java
boolean locked = false;

try {
    lock.lockInterruptibly();
    locked = true;

    // 获取锁成功后的业务代码
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
} finally {
    if (locked) {
        lock.unlock();
    }
}
```

特点：

>在等待锁期间收到中断信号时，会抛出InterruptedException，线程可以及时结束任务

## 6. ReentrantLock 与 synchronized

| 对比项 | `synchronized`     | `ReentrantLock` |
| --- |--------------------|-----------------|
| 锁类型 | JVM内部锁，锁对象的Monitor | 显示锁             |
| 获取锁 | 有                  | 有               |
| 释放锁 | 离开同步代码块后自动释放锁                  | 必须手动调用 unlock()，通常放在 finally 中               |
| 可重入 | 有                  | 有               |
| 尝试获取锁 | 无                  | 有               |
| 超时获取锁 | 无                  | 有               |
| 可中断获取锁 | 无                  | 有               |
| 公平锁 | 无                  | 有               |
| 条件队列 | 无                  | 有               |

我的选型结论：

>简单临界区优先使用synchronized，需要tryLock避免无限等待、需要可中断地获取锁、需要超时机制、需要公平锁、需要多个Condition精细协调线程选择ReentrantLock

## 8. 高频面试题

### 8.1 ReentrantLock 是什么？为什么出现？

>ReentrantLock是显示锁，synchronized只能处理简单的临界区线程安全，不能释放锁，ReentrantLock可释放锁

### 8.2 lock() 和 unlock() 之间能保证原子性、可见性、有序性吗？

>可以

### 8.3 为什么 unlock() 必须放在 finally 中？

>避免在执行业务代码的过程中抛出异常代指2没有释放锁

### 8.4 ReentrantLock 的可重入性是什么？

>已经获得锁的线程在调用方法时无需再竞争锁，获得多少次锁就要释放多少次锁

### 8.5 tryLock()、超时 tryLock()、lockInterruptibly() 有什么区别？

>tryLock()获取锁失败后立即返回flase，超时tryLock()在指定时间内获取锁失败后返回false,线程在等待lockInterruptibly()获取锁时收到中断，会抛出InterruptedException并放弃等待； 它不会自动释放已经持有的锁，已持有的锁仍必须在 finally 中 unlock()。

### 8.6 ReentrantLock 与 synchronized 如何选择？

>简单临界区优先使用synchronized，需要tryLock避免无限等待、需要可中断地获取锁、需要超时机制、需要公平锁、需要多个Condition精细协调线程选择ReentrantLock

## 9. 算法题：用栈实现队列

题目：

```text
仅使用两个栈，实现队列的：
push(x)
pop()
peek()
empty()
```

我的初始解法：

>A栈负责存放原始顺序的元素，B栈负责存放队列顺序的元素，每次push()都先把元素放到A栈，再遍历A栈把元素放到B栈

优化思路：

```text
in 栈：负责入队
out 栈：负责出队
仅当 out 为空且需要 pop/peek 时，
才把 in 中全部元素转移到 out。
```

为什么不在每次 `push()` 时搬运元素：

>时间复杂度会变成O(n2)

时间复杂度：

| 操作 | 复杂度  |
| --- |------|
| `push()` | O(1) |
| `pop()` |   均摊 O(1)   |
| `peek()` |  均摊 O(1)    |
| `empty()` | O(1)     |

代码：

```java
class MyQueue {
    private final Deque<Integer> in = new ArrayDeque<>();
    private final Deque<Integer> out = new ArrayDeque<>();

    public void push(int x) {
        in.push(x);
    }

    public int pop() {
        moveIfNeeded();
        return out.pop();
    }

    public int peek() {
        moveIfNeeded();
        return out.peek();
    }

    public boolean empty() {
        return in.isEmpty() && out.isEmpty();
    }

    private void moveIfNeeded() {
        if (out.isEmpty()) {
            while (!in.isEmpty()) {
                out.push(in.pop());
            }
        }
    }
}
```
